# uvpacket worked example + comprehensive abstraction-leak fixes

## Context

Adding `uvpacket` as a worked-example protocol exposed that
`mfsk-core`'s "generic over `P: Protocol`" pipeline silently
hardcodes a great many Wsjt77 / FT8 family assumptions. Two leaks
have been fixed (CRC-14 inside `bp_decode` → moved to
`MessageCodec::verify_info`, committed `014892e`); a code audit
identified six more. Until they're addressed, `uvpacket` cannot
use the existing `pipeline::decode_frame::<P>` and the example
ceases to demonstrate the library's reusable infrastructure.

This plan sequences the abstraction-leak fixes in order, then lands
uvpacket on top of the cleaned pipeline, then the
`docs/ADDING_A_PROTOCOL.md` walkthrough and the `signed-qsl`
application example. Result: a Phase 0.3.0 release where every
LDPC-based protocol uses one shared pipeline, and the trait surface
is genuinely the only place each protocol needs to declare its
specifics.

## Bespoke-path leaks found (additional audit)

A follow-up sweep covering Q65, WSPR, JT9, JT65 (the protocols that
do NOT use `pipeline::decode_frame::<P>`) and the FEC sub-layers
found four more leaks beyond the eight table-listed below:

| | Site | Leak | Disposition for 0.3.0 |
|---|------|------|----------------------|
| **A** | `fec/qra/q65.rs::Q65Codec::decode` | Hardcoded CRC-12 verify inside the Q65 codec — same root cause as the LDPC CRC-14 leak that Phase 0 fixed | **Migrate to `Q65Message::verify_info`** as a posterior commit after Phase 0; ~30 lines, keeps the codec-internal CRC implementation but exposes the abstraction |
| **B** | `fec/ldpc/bp.rs::BpResult.message77: [u8;77]` | Internal LDPC result struct duplicates the Wsjt77-shaped 77-bit field; same shape as `DecodeResult.message77` (leak #3) | **Land with Phase B** — change to `info: [u8; LDPC_K]`, callers slice `[..77]` for Wsjt77-based protocols |
| **C** | `msg/ap.rs::ApHint { call1, call2, grid, report }` | Struct shape assumes callsign-based hints; Q65 already needs `ap_hint_to_q65_mask` to translate, uvpacket / future byte-payload protocols can't use it | **Phase C marker trait** — `ApHint::build_bits<P>` requires `WsjtApCompatible` (Wsjt77Message + Q65Message implement); uvpacket-class protocols define their own AP scheme later |
| **D** | Result-type fragmentation (`DecodeResult`, `Q65Decode`, `WsprDecode`, `Jt9Decode`, `Jt65Decode`) | No unified `Decode<P>` — each bespoke path returns its own shape | **Out of scope** — different protocols carry different per-decode metadata (Q65 iterations, WSPR power_dbm, …); defer unification to 0.4.0+ |
| **E** | Duplicated frame-search glue across bespoke paths | Same outer pattern (`audio → coarse search → per-candidate decode`) in `q65::rx`, `wspr::rx`, `jt9::rx`, `jt65::rx`; each rewrites the loop | **Out of scope** — structural differences (slot vs no-slot, block vs interleaved sync, sub-mode dispatch) make a single unified loop infeasible |

### FEC layer cleanliness check

The convolutional family (`ConvFano` for WSPR, `ConvFano232` for
JT9) does **not** carry an internal CRC: Fano sequential decode's
own terminal-state condition serves as integrity. Reed-Solomon
(`Rs63_12` for JT65) is self-checked via its syndrome. Only the
LDPC family had the CRC-baked-in problem (Phase 0 fixed it), plus
the QRA codec used by Q65 (leak A above, post-Phase 0 migration).

## Audit summary — eight pipeline-surface leaks found

| # | Location | Hardcoded assumption | Fix phase |
|---|----------|----------------------|-----------|
| 1 | `bp_decode` / OSD CRC-14 (FIXED `014892e`) | CRC-14 inside FEC | Phase 0 |
| 2 | `pipeline.rs::encode_tones_for_snr<P>(msg77: &[u8;77], …)` (line 279–312) | Bakes CRC-14/24 from callsign bits | Phase A |
| 3 | `DecodeResult.message77: [u8;77]` (line 73–84) | Caller-visible 77-bit field | Phase B |
| 4 | `decode_frame` dedup `x.message77 == r.message77` (line 430, 510) | Exact-match on 77-bit | Phase B |
| 5 | `decode_frame_subtract` SIC re-encode via `encode_tones_for_snr` (line 519) | Inherits leak #2 | Phase A |
| 6 | `pipeline_ap.rs` module (line 1) | Assumes `P::Msg = Wsjt77Message` | Phase C |
| 7 | `ApHint::build_bits` hardcodes Wsjt77 bit layout (call1 at 0–28 etc.) | Wsjt77 field positions | Phase C |
| 8 | `mfsk-ffi/src/lib.rs:339,354,571` reads `.message77` + `unpack77` | FFI wired to `[u8;77]` | Phase B |

The DSP / sync / LLR / tx / subtract / equaliser layers
(`coarse_sync<P>`, `refine_candidate<P>`, `symbol_spectra<P>`,
`compute_llr<P>`, `codeword_to_itone<P>`, `equalize_local<P>`,
`subtract_tones`, `compute_snr_db<P>`) are clean — they operate on
the trait-level constants only. The leaks are concentrated in
**three places**: the pipeline result type, the SNR re-encoder, and
the AP-assisted module.

## Phase plan

Each phase is a self-contained, atomic commit. Tests stay green
across every phase; only `mfsk-ffi` migrates in lockstep with the
result-shape change in Phase B.

### Phase 0 — CRC abstraction (DONE — `014892e`)

`MessageCodec::verify_info` default returns `true`; `Wsjt77Message`
overrides to delegate to `check_crc14`; `bp_decode` /
`osd_decode_*` accept `Option<fn(&[u8]) -> bool>` and stop hardcoding
CRC-14. FT8/FT4/FST4 byte-identical behaviour confirmed.

### Phase A — Replace `encode_tones_for_snr`'s 77-bit hack with FEC re-encode

**Problem**: The function rebuilds a 91-bit (FT8/FT4) or 101-bit
(FST4) info vector by appending a hand-coded CRC-14 / CRC-24 to a
77-bit message. That hand-coded CRC logic is duplicated from the
FEC layer, hardcodes the WSJT message format, and breaks for any
protocol whose 91 info bits aren't structured as 77 message + 14
CRC.

**Fix**: Take `info: &[u8]` (length `<P::Fec as FecCodec>::K`)
directly. Re-encode via `fec.encode(info, &mut codeword)` →
`codeword_to_itone::<P>(&codeword)`. Symmetric with how the
encoder side already works; no message-format knowledge needed.
Caller (pipeline) switches from `&r.message77` to `&r.info[..]`.

**Files**:
- `mfsk-core/src/core/pipeline.rs:279–312` — replace function body
- `mfsk-core/src/core/pipeline.rs:158, 191, 219, 519` — caller updates
- `mfsk-core/src/msg/pipeline_ap.rs` — caller updates (also see Phase C)

**Test**: existing FT8/FT4/FST4 SNR-sweep tests pass with
byte-identical SNR_DB outputs.

### Phase B — `DecodeResult.message77` → `info: Vec<u8>`

**Problem**: result struct hardcodes 77-bit message length. uvpacket
needs to ship 91 bits (4-bit length + 80-bit byte payload + 7-bit
pad); future protocols may need other widths.

**Fix**:
- `DecodeResult.message77: [u8; 77]` → `info: Box<[u8]>` of length
  `<P::Fec as FecCodec>::K` (using `Box<[u8]>` rather than `Vec<u8>`
  preserves `Clone + PartialEq`-friendly semantics for the dedup
  loop and avoids reallocating).
- Add helper `DecodeResult::message77(&self) -> &[u8]` returning
  `&self.info[..77]` — small ergonomic backward-compat for
  `mfsk-ffi` and tests during migration.
- `decode_frame` dedup: compare `info` fields directly (`Box<[u8]>`
  derives PartialEq).

**Files**:
- `mfsk-core/src/core/pipeline.rs:73–84` — struct definition
- `mfsk-core/src/core/pipeline.rs:155–230, 430, 510, 519` — usages
- `mfsk-core/src/msg/pipeline_ap.rs` — finalise_result construction
- `mfsk-ffi/src/lib.rs:339, 354, 571` — FFI consumers switch to
  `&result.info[..77]` for the byte-array argument; the rest of
  the FFI byte path (FT8/FT4/FST4 → `unpack77`) is unchanged.

**Test**: all FT8/FT4/FST4 unit + integration tests pass; cpp_smoke
roundtrips still work.

### Phase C — Mark `pipeline_ap.rs` as Wsjt77-family-only

**Problem**: Module assumes `P::Msg = Wsjt77Message` (callsign /
grid / report at fixed bit positions). uvpacket's
`PacketBytesMessage` carries arbitrary bytes — call1 at bits 0–28
collides with PacketBytes' length field at 0–3.

**Fix** (minimum): add a marker trait `WsjtApCompatible` that
`Wsjt77Message` implements. Bound `process_candidate_ap<P>` on
`<P::Msg as MessageCodec>: WsjtApCompatible`. This keeps the
existing FT8/FT4/FST4 AP path working unchanged and makes the
restriction explicit at the type level.

uvpacket does **not** offer an AP-biased BP path in this milestone
— a future revision could generalise `ApHint` to be
protocol-defined, but that's out of scope for the example.

**Files**:
- `mfsk-core/src/msg/ap.rs` — add marker trait + impl
- `mfsk-core/src/msg/pipeline_ap.rs` — add bound
- (No callers break — FT8/FT4/FST4 already use Wsjt77Message)

### Phase 1 — `PacketBytesMessage` codec (DONE — `b585f9d`)

Already committed. 4-bit length + 80-bit payload + 7-bit pad,
`MessageCodec::PAYLOAD_BITS = 91`, `Unpacked = Vec<u8>`. No CRC
override (default `verify_info` returns true).

### Phase 2 — `uvpacket` module proper

Now possible because Phases A + B mean
`pipeline::decode_frame::<UvPacket300>` works without further
refactor.

**New files**:
- `mfsk-core/src/uvpacket/mod.rs` — module docstring + re-exports
- `mfsk-core/src/uvpacket/protocol.rs` — `uvpacket_submode!` macro
  emitting `UvPacket150` / `UvPacket300` / `UvPacket600` ZSTs
  (4-FSK GFSK, h=1.0, BT=1.0, NDOWN=2). All three share
  `type Fec = Ldpc174_91`, `type Msg = PacketBytesMessage`,
  `const ID = ProtocolId::UvPacket`.
- `mfsk-core/src/uvpacket/sync_pattern.rs` — single Costas-4 block
  `[0, 1, 3, 2]` at symbol 0 (reused from FT4_COSTAS_A).
- `mfsk-core/src/uvpacket/chain.rs` — frame-chain layer for
  >10-byte payloads (~120 lines, already drafted in working tree
  on `main`).
- `mfsk-core/src/uvpacket/tx.rs` — encode bytes via
  `PacketBytesMessage::pack` → `Ldpc174_91::encode` →
  `codeword_to_itone::<P>` → GFSK synth. Per-sub-mode `DownsampleCfg`
  built from trait constants.
- `mfsk-core/src/uvpacket/rx.rs` — wraps `pipeline::decode_frame::<P>`
  + `PacketBytesMessage::default().unpack(&result.info[..91], ctx)`
  for the typed `Vec<u8>` return. Multi-signal in band is automatic
  (pipeline's `coarse_sync` already finds parallel candidates).

**Wiring**:
- `mfsk-core/src/lib.rs` — `pub mod uvpacket;` gated on `feature = "uvpacket"`; reexport ZSTs.
- `mfsk-core/src/registry.rs` — three `protocol_meta!` lines.
- `mfsk-core/tests/protocol_invariants.rs` — three
  `assert_protocol_invariants::<UvPacket*>` lines.

**Tests**:
- `mfsk-core/tests/uvpacket_roundtrip.rs` — synth → decode roundtrip
  per sub-mode, byte payloads of length 1, 5, 10.
- `mfsk-core/tests/uvpacket_multi_signal.rs` — two simultaneous
  frames at different audio centres in one buffer; both decoded.

### Phase 4 — `examples/signed-qsl/` crate + `docs/APPLICATION_SIGNED_QSL.{md,ja.md}`

Standalone workspace crate (`publish = false`). k256 ECDSA over
secp256k1 + serde-bincode for compact QSL serialisation. Demonstrates
how a real application stacks on uvpacket:

- `QslMessage` — callsigns, frequency, mode, time, report (~50–100 B)
- `SignedQsl` — `QslMessage` + 64-byte Schnorr sig + 8-byte pubkey
  hash (~120–180 B total)
- Encode pipeline: pack → sign → bincode → chain-encode → uvpacket synth
- Decode pipeline: uvpacket decode → chain reassemble → bincode → verify

Doc explains motivation (off-grid QSL, no Internet/postal mail
required) and the migration path from existing AX.25 + ECDSA TNC
implementations (the application bytes are unchanged; only the
modem swaps).

## Devel-branch landing strategy

Phases A through 5 are big enough to warrant a `devel` branch
rather than landing directly on `main`. The first action after
exiting plan mode:

1. Create `devel` from current `main` HEAD (`014892e`).
2. Stage the currently-uncommitted uvpacket skeleton work
   (`mfsk-core/src/uvpacket/{mod,protocol,sync_pattern,chain,tx}.rs` —
   already on disk, untracked) as a "WIP: uvpacket skeleton" commit
   on `devel`. The skeleton compiles in isolation but isn't yet
   wired into `lib.rs` because the abstraction-leak fixes (Phases
   A + B) need to land first.
3. Push `devel` to origin so the work is shareable / reviewable
   before merging back to `main`.
4. Subsequent phases (A, B, C, 2, 3, 4, 5) land on `devel` as
   atomic commits. When `devel` is fully green and the user is
   satisfied, fast-forward `main` and tag `v0.3.0`.

## Out of scope (explicitly)

- AP-biased BP for uvpacket — Phase C marks the AP module as
  Wsjt77-only. uvpacket-specific AP can come later via a generic
  ApHint design.
- Generalised `pipeline_ap` — same reason.
- mfsk-ffi exposure of uvpacket — additive when needed; the plan
  ships uvpacket as a Rust-API protocol first.
- Higher-layer AX.25 / KISS / APRS framing — caller responsibility.
- Production-grade key directory / cert revocation in signed-qsl —
  example uses a static `PublicKeyDirectory`.
