# uvpacket worked example + reframed abstraction-leak fixes

## Context

Adding `uvpacket` as a worked-example protocol exposed that
`mfsk-core`'s "generic over `P: Protocol`" pipeline silently
hardcodes a great many Wsjt77 / FT8 family assumptions. Two leaks
have been fixed (CRC-14 inside `bp_decode` → moved to
`MessageCodec::verify_info`, committed `014892e`); a code audit
identified more.

The original plan framed the rest as "make `decode_frame::<P>`
generic over arbitrary `<P::Fec as FecCodec>::K`". Re-examination
revised the framing on two axes:

1. **LDPC matrices are realistically a closed set.** `Ldpc174_91`
   and `Ldpc240_101` are both hand-crafted for very short
   blocklengths; adding a third is a research project, not a
   refactor. **But** the BP / OSD / encode algorithms are
   identical across the two sizes — `fec/ldpc/bp.rs` and
   `fec/ldpc240_101/bp.rs` are ~95% duplicate code differing only
   in tables and constants. The honest abstraction is therefore
   *abstract the parameters, keep the algorithm, enforce the
   closed set at compile time via a sealed trait*.

2. **uvpacket with no CRC is over-aggressive.**
   `PacketBytesMessage` currently uses the default `verify_info`
   (always accept), so any parity-converged BP candidate yields
   garbage to the application with no integrity signal. The 7
   trailing pad bits in the existing 91-bit layout should carry a
   CRC-7.

The reframed shape:

- **Single generic LDPC codec** parameterised over `LdpcParams`
  (sealed). One BP, one OSD, one encode. Algorithmic improvements
  land once and apply to both protocols. ~600 lines of duplicated
  `ldpc240_101/{bp,osd}.rs` go away.
- **Pipeline trusts `FecResult.info`** end-to-end: pass it
  straight to `fec.encode(&info)` for SNR re-encoding, no CRC
  reconstruction in the pipeline. ~100 lines of duplicated
  `crc14`/`crc24` helpers in `pipeline.rs` + `pipeline_ap.rs` go
  away.
- **`DecodeResult.info` becomes `Box<[u8]>`** so uvpacket's
  91-bit-non-Wsjt77 info rides the same pipeline as FT4/FST4.
- **`Wsjt77Message::verify_info` length-dispatches** between
  CRC-14 (K=91) and CRC-24 (K=101) — fixes a Phase-0 leak where
  FST4 currently relies on the FEC short-circuiting `verify_info`
  internally.
- **Q65 CRC-12 verification** moves from `Q65Codec::decode` into
  `Q65Message::verify_info` — same shape as Phase 0 for LDPC.
- **`PacketBytesMessage::verify_info` adds CRC-7** in the trailing
  pad bits.
- **AP module is gated to `WsjtApCompatible`** message codecs.

## Hidden Phase-0 gaps (must close first)

Audit revealed that commit `014892e` (CRC-14 abstraction for
`Ldpc174_91`) was only one of three FEC layers needing the same
treatment:

- **`Ldpc240_101::bp_decode` / `osd_decode*`** do **not** accept a
  `verify` parameter — they hardcode `check_crc24` internally
  (`fec/ldpc240_101/bp.rs:139`).
- **`Wsjt77Message::verify_info`** hardcodes `check_crc14` and
  rejects any slice whose length ≠ 91 (`msg/mod.rs:99-101` +
  `fec/ldpc/bp.rs:46-48`). FST4 (K = 101) currently survives only
  because the FEC's internal short-circuit accepts the codeword
  before `verify_info` is consulted.
- **`Q65Codec::decode`** hardcodes the CRC-12 / CRC-6 verification
  internally (`fec/qra/q65.rs:288-302`). The `Q65Fec` `FecCodec`
  impl is a stub that returns `None` (`q65/protocol.rs:209-214`)
  because Q65 uses GF(64) symbol BP, not bit-LLR — Q65 actually
  decodes via the bespoke `q65/rx.rs` path that calls
  `Q65Codec::decode` directly. The CRC integrity check belongs on
  `Q65Message::verify_info`.

Phase 0c absorbs the LDPC half (both variants share one BP that
honours the trait-level `verify_info`); Phase 0d does the same
for QRA / Q65.

The other FECs (`ConvFano`, `ConvFano232`, `Rs63_12`) carry no
CRC — Fano sequential decode's terminal-state condition is the
integrity check, Reed-Solomon syndrome verification is built into
`Rs63_12::decode_soft`. No leak there.

## Phase plan

Each phase is a self-contained, atomic commit. Tests stay green
across every phase; only `mfsk-ffi` migrates in lockstep with the
result-shape change in Phase B.

### Phase 0 — CRC abstraction, LDPC174_91 only (DONE — `014892e`)

`MessageCodec::verify_info` default returns `true`; `Wsjt77Message`
overrides to delegate to `check_crc14`; `bp_decode` /
`osd_decode_*` accept `Option<fn(&[u8]) -> bool>` and stop hardcoding
CRC-14. FT8/FT4/FST4 byte-identical behaviour confirmed.

### Phase 0c — Unify LDPC under sealed `LdpcParams` trait

#### Shape (as landed)

```rust
// mfsk-core/src/fec/ldpc/params.rs
mod sealed { pub trait Sealed {} }

pub trait LdpcParams: sealed::Sealed + Copy + Default + 'static {
    const N: usize;
    const K: usize;
    const M: usize;
    const MAX_ROW: usize;
    fn mn(bit: usize) -> [u8; 3];
    fn nm(check: usize, slot: usize) -> u8;
    fn nrw(check: usize) -> u8;
    fn gen_parity(row: usize, col: usize) -> u8;
}

#[derive(Copy, Clone, Debug, Default)]
pub struct Ldpc174_91Params;
impl sealed::Sealed for Ldpc174_91Params {}
impl LdpcParams for Ldpc174_91Params { /* delegate to fec::ldpc::tables + osd::GEN_PARITY */ }

#[derive(Copy, Clone, Debug, Default)]
pub struct Ldpc240_101Params;
impl sealed::Sealed for Ldpc240_101Params {}
impl LdpcParams for Ldpc240_101Params { /* delegate to fec::ldpc240_101::tables */ }
```

The generic algorithm lives in [`fec::ldpc::bp::bp_decode_generic<P>`]
and [`fec::ldpc::osd::{osd_decode_generic<P>, ldpc_encode_generic<P>}`].
Both `Ldpc174_91` and `Ldpc240_101` keep their own `FecCodec` impls
as thin wrappers that pin the generic to the appropriate params type
(per-protocol behaviour like Ldpc240_101's `osd_depth.min(3)` cap
stays explicit at the wrapper level rather than being smuggled into
a trait constant). Sealing keeps the closed-set property — external
crates cannot add new params types.

#### Generic algorithm

`bp_decode_generic<P: LdpcParams>` operates on `Vec<f32>` working
buffers sized from `P::N`, `P::M`, `P::MAX_ROW` (heap, ~3 KB at
most; negligible vs the 30+ BP iterations). `osd_decode_generic<P>`
uses a `Vec<u8>` flat permuted-generator matrix (`P::K * P::N` =
~24 KB worst case) and reuses per-order candidate buffers across
iterations to avoid per-candidate allocs.

`BpResult` and `OsdResult` carry both `info: Vec<u8>`
(length P::K) and `codeword: Vec<u8>` (length P::N), plus a
fixed-size `message77: [u8; 77]` for backward-compat with FT8's
bespoke decode path which reads it directly.

#### Files

- New: `mfsk-core/src/fec/ldpc/params.rs`.
- Modified: `mfsk-core/src/fec/ldpc/{mod,bp,osd}.rs` — generic over
  `LdpcParams`. `pub type Ldpc174_91 = LdpcCodec<Ldpc174_91Params>`.
- Modified: `mfsk-core/src/fec/ldpc/tables.rs` — add zero-padded
  `NM_FLAT` and `GEN_PARITY_FLAT`.
- Deleted: `mfsk-core/src/fec/ldpc240_101/{bp,osd}.rs` (~600 lines).
- Modified: `mfsk-core/src/fec/ldpc240_101/{mod,tables}.rs` —
  reduces to ~30 lines: re-exports + CRC-24 helpers.
- Modified: `mfsk-core/src/msg/mod.rs` — `Wsjt77Message::verify_info`
  length-dispatches:

  ```rust
  fn verify_info(info: &[u8]) -> bool {
      match info.len() {
          91  => crate::fec::ldpc::check_crc14(info),
          101 => crate::fec::ldpc240_101::check_crc24(info),
          _   => false,
      }
  }
  ```

#### Acceptance

- `cargo test -p mfsk-core --features full --release` — green.
- FT4 + FST4 SNR-sweep tests **byte-identical** vs pre-Phase-0c.
- ~600 LOC reduction in `mfsk-core/src/fec/`.

### Phase 0d — Q65 CRC migration to `Q65Message::verify_info`

Mirror Phase 0 for the QRA / Q65 codec.

- `mfsk-core/src/fec/qra/q65.rs` — `Q65Codec::decode` becomes
  CRC-agnostic; the trailing-CRC check is dropped from this layer.
  `Q65DecodeError::CrcMismatch` retained as type but not produced
  internally.
- `mfsk-core/src/msg/q65.rs::Q65Message::verify_info` — override
  the default `true` impl to delegate to a `q65_verify_crc12`
  helper that:
  1. Repacks the 78-bit `info` slice into 13 GF(64) symbols.
  2. Calls the existing `crc12()` over the user portion.
  3. Compares to the trailing two CRC symbols.
- `mfsk-core/src/q65/rx.rs` — after `Q65Codec::decode` returns,
  call `Q65Message::verify_info(&decoded_bits)`; on `false`,
  reject the candidate.

#### Acceptance

- Q65 SNR-sweep + roundtrip tests stay byte-identical.
- The Q65Codec layer is CRC-agnostic and reusable for AP / list-
  decoding paths that may want to bypass the CRC.

### Phase A — Delete pipeline-side CRC reconstruction

Now that both LDPC variants honour `verify_info`, `r.info[..]`
satisfies `info[77..K] == crc(info[..77])` whenever the verifier
accepts. Therefore `fec.encode(&r.info)` reproduces the same
codeword as today's "extract msg77, recompute CRC, encode" path
— bit-identical.

```rust
fn encode_tones_for_snr<P: Protocol>(info: &[u8], fec: &P::Fec) -> Vec<u8> {
    let mut cw = vec![0u8; P::Fec::N];
    fec.encode(info, &mut cw);
    codeword_to_itone::<P>(&cw)
}
```

The `match P::Fec::K { 91 => crc14, 101 => crc24, _ => zeros }`
branch and the local `crc14` / `crc24` helpers go away entirely.

#### Files

- `mfsk-core/src/core/pipeline.rs:279–366` — replace function body,
  delete local CRC helpers.
- `mfsk-core/src/core/pipeline.rs:163-164, 197-198, 225-226, 519` —
  call sites pass `&r.info` directly.
- `mfsk-core/src/msg/pipeline_ap.rs:276-289, 303-318` —
  `finalise_result` drops local CRC reconstruction; `crc14_local`
  removed.

#### Acceptance

- FT4 + FST4 SNR-sweep tests byte-identical.


### Phase D — uvpacket CRC-7

Replace the 7 trailing zero-pad bits with a CRC-7 over the head
84 bits. Polynomial `x^7 + x^3 + 1` (0x09, SD-card standard).
False-accept rate ≈ 1/128 = 0.78% on random parity-converged
garbage; combined with LDPC's already-low post-FEC BER, false
decodes drop ~2 orders of magnitude vs the current
"always accept".

```text
bits  0 ..  4 : length (4 bits)
bits  4 .. 84 : 10 bytes payload (80 bits)
bits 84 .. 91 : CRC-7 over bits 0..84
```

#### Files

- `mfsk-core/src/msg/packet_bytes.rs` — add `fn crc7(head: &[u8])
  -> u8`. `pack` fills bits 84..91 with CRC-7. `unpack` validates
  it; `MessageCodec::verify_info` overrides the default to do the
  same check (BP rejects mid-iteration on CRC-7 fail).
- New tests: roundtrip + single-bit-flip rejection.

uvpacket isn't released yet on `main`, so no on-the-wire
compatibility concern.

## Out of scope (explicitly)

- **Audit row D — Result-type fragmentation across bespoke paths.**
  `DecodeResult` (FT4/FST4/uvpacket via pipeline), `Q65Decode`,
  `WsprDecode`, `Jt9Decode`, `Jt65Decode` each carry different
  per-decode metadata. Unifying them needs tagged enum / `dyn
  Decode` / GAT — none obviously right. Defer to 0.4.0+.
- **Audit row E — Duplicated frame-search glue across bespoke
  paths.** Each of `q65/rx.rs`, `wspr/rx.rs`, `jt9/rx.rs`,
  `jt65/rx.rs` repeats *audio → coarse search → per-candidate
  decode*. Sharing requires solving (D) plus reconciling
  structural differences. Defer.
- **Reed-Solomon / convolutional unification.** Single concrete
  code each (RS(63,12) for JT65, ConvFano variants for WSPR/JT9).
  No parallel implementations to share.
- **Further refactor of the bespoke FT8 decode path.** FT8 has its
  own rx loop that doesn't use `pipeline::decode_frame::<Ft8>`;
  staying out of it isolates risk for 0.3.0.
- **AP-biased BP for uvpacket** — Phase C marks the AP module as
  Wsjt77-only. uvpacket-specific AP can come later via a generic
  ApHint design.
- **Generalised `pipeline_ap`** — same reason.
- **mfsk-ffi exposure of uvpacket** — additive when needed; the
  plan ships uvpacket as a Rust-API protocol first.
- **Higher-layer AX.25 / KISS / APRS framing** — caller responsibility.
- **Production-grade key directory / cert revocation in
  signed-qsl** — example uses a static `PublicKeyDirectory`.
